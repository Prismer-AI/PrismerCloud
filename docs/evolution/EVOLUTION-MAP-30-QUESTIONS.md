## 一、用户是谁、为什么来

### Q1. 这个页面的主要用户是谁？

**研究了什么：** 看了 CLAUDE.md 的 Audience 定义、evolution page 的访问路径、auth 要求。
**质疑：** 当前 MAP-DESIGN 假设用户是"已有 Agent 的开发者"，但页面是公开的（不需要登录）。
**结论：**

三种用户：

1. **路过的访客**（最多）——从 landing page 点进来，还没有 Agent，想看"这个系统在干什么"
2. **已有 Agent 的开发者**——想看自己的 Agent 学到了什么，应该装哪个 Gene
3. **平台运营者**——想看全局进化健康度

**修改 MAP PRD：** 用户画像从"已有 Agent"改为"三层用户"，冷启动页面首先服务访客。

### Q2. 用户来这个页面的动机是什么？

**研究了什么：** 分析了 /evolution 页面在导航栏的位置（与 Playground、Docs 同级），以及 4 个 tab（Map/Library/Feed/My Evolution）的分工。
**质疑：** Map 试图同时满足"看全貌"和"看细节"，但这两个需求的交互模式完全不同。
**结论：**

Map 不是 Dashboard，不是操作台。Map 的唯一动机是：**"让我一眼看到进化正在发生"**。

- 访客：好奇这个系统长什么样 → 需要视觉震撼
- 开发者：我的 Agent 今天学了什么 → 需要定位到"我"
- 运营者：系统健不健康 → 需要全局统计

**修改 MAP PRD：** 核心价值从"系统认知状态的实时投影"收窄为"让人一眼看到进化正在发生"。

### Q3. 用户打开页面后前 3 秒想看到什么？

**研究了什么：** 当前实际效果——45 个 seed gene 散落一屏，没有焦点，没有故事，没法看。
**质疑：** 当前设计说"定位到最近事件的 L1 级别"，但现实是没有最近事件（冷启动），所以 fallback 到全图。
**结论：**

前 3 秒必须看到的：

- **有焦点的画面**——不是一堆等大的点，而是一个明显的"主角"
- **一句话故事**——"刚才 agent-X 用 Timeout Recovery 解决了 error:timeout"
- **动态感**——粒子在流、脉冲在闪、不是一张静态截图
- 如果没有真实数据：**一个引导性的 CTA**，不是一个空荡荡的图

**修改 MAP PRD：** 冷启动不画图。改为故事卡片 + 精选 Gene 展示 + CTA。

### Q4. 用户有没有可能是潜在用户（还没有 Agent）？

**结论：** 大概率是。evolution 页面是公开的，从 landing page 一键可达。
**修改 MAP PRD：** 冷启动策略从"画虚线图"改为"展示价值主张 + 精选 Gene 卡片 + 注册 CTA"。

### Q5. 用户打开页面时是否期望看到自己的数据？

**结论：** 未登录用户不期望。已登录用户在 "My Evolution" tab 里看自己的数据。Map tab 是全局视角。
**修改 MAP PRD：** Map 永远是公开全局视角，不按用户过滤。

## 二、数据现状与冷启动

### Q6. 当前有多少真实进化数据？

**研究了什么：** 查了 dev DB——45 个 seed gene，0 个真实 capsule（测试数据除外），73 条 edge（全是 seed 隐含的 0-0 边）。
**结论：** 零。一切都是冷启动。Graph 在没有数据时是一堆等大的圆和虚线——完全没有意义。

### Q7. Seed data 有什么意义？

**质疑：** 当前设计把 45 个 seed gene 当作"预连线的节点"画出来。但 seed gene 没有执行数据、没有成功率、没有故事。画出来只是噪音。
**结论：** Seed gene 的价值不在于可视化，而在于**推荐**——当用户真正遇到问题时，系统能推荐一个 seed gene。在 Map 上画 45 个空圆圈不传递任何信息。

**修改 MAP PRD：** 冷启动不画 seed gene 图。改为"精选 Gene 卡片列表"（从 seed gene 中按 category 选 6-9 个最有代表性的展示）。

### Q8. 冷启动状态下 Map 应该展示什么？

**结论（核心决策）：**

冷启动 ≠ 空图。冷启动 = **故事模式（Story Mode）**：

```
┌──────────────────────────────────────────────────────────────┐
│  🧬 Evolution Map                              [Get Started] │
│                                                               │
│  "Agents learn from success and failure.                     │
│   Every execution makes the network smarter."                │
│                                                               │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐                  │
│  │ ⬡ Repair │  │ ○ Optimize│  │ ◇ Innovate│                 │
│  │          │  │          │  │          │                    │
│  │ Timeout  │  │ Rate     │  │ Code     │                   │
│  │ Recovery │  │ Limit    │  │ Refactor │                   │
│  │          │  │ Backoff  │  │ Agent    │                   │
│  │ 0 runs   │  │ 0 runs   │  │ 0 runs   │                  │
│  │ [Install]│  │ [Install]│  │ [Install]│                   │
│  └──────────┘  └──────────┘  └──────────┘                   │
│                                                               │
│  "Install a gene → run your agent → watch the map light up" │
└──────────────────────────────────────────────────────────────┘
```

不画图。画卡片。

### Q9. 当有真实数据时，什么数据最有故事感？

**研究了什么：** EvolutionStory 数据结构（谁/什么 gene/干了什么/效果）。
**结论：** 最有故事感的数据是**一条最近的 capsule**——"agent-X 遇到 error:timeout，用 Timeout Recovery 策略，重试 2 次后成功，成功率从 76% 提升到 79%"。这比任何图都直观。

### Q10. 多久会产生一条真实数据？

**结论：** 取决于有多少活跃 Agent。当前（未部署到生产）是零。部署后，每个使用 evolution SDK 的 Agent 每次 analyze+record 产生一条。可能每天几条到几百条。

## 三、信息层级与展示形式

### Q11. 一条"进化路径"对用户意味着什么？

**结论：** "error:timeout → Timeout Recovery = 79% 成功率"——这条路径告诉用户"遇到 timeout，用这个策略有 79% 概率解决"。这是一个**推荐**，不是一个图谱节点。

### Q12. 什么是一个"领域"（Domain/Cluster）？

**质疑：** 当前设计用 Louvain 社区检测 + SignalTag.type 前缀来定义 domain。但用户不关心"这个 gene 在哪个 cluster"——用户关心"这个 gene 能解决我的问题吗"。
**结论：** Domain 在 MAP 可视化中的价值是**视觉分组**，不是业务概念。可以简化为按 gene category（repair/optimize/innovate）分区，不需要 Louvain。

### Q13. "局部"和"整体"应该用什么方式展示？

**质疑：** 当前设计用 4 级缩放（L1-L4 宇宙隐喻）。但缩放本身就是一个高认知成本的操作——用户需要先理解"这是一个可缩放的画布"才能使用。
**结论：** 不缩放。两种模式足够：

1. **故事模式（默认）**——卡片列表，一眼看最近发生了什么
2. **图谱模式（手动切换）**——数据足够多时才有意义的 Graph

### Q14. 用户需要同时看到多少信息？

**结论：** 最多 3-5 条最近事件 + 3-6 个推荐 Gene。不需要同时看到 45 个 gene 和 60 个 signal。

### Q15. 图（Graph）是最好的展示形式吗？

**质疑：** 这是最核心的问题。当前实现是一个力导向图——但它在冷启动时是一堆等大的圆和虚线，没有层次、没有故事、没有引导。
**结论：** Graph **不是**冷启动的最佳展示。Graph 只在有足够数据（≥50 真实 capsules）时才有意义。在此之前，用**故事卡片 + 推荐列表**。

**修改 MAP PRD：** 引入"自适应模式"——数据少时用卡片模式，数据多时自动切换到图模式。

### Q16. 有没有更适合冷启动的展示形式？

**结论：** 有。三种候选：

1. **故事卡片流**——类似 Feed，但只显示精选的 1-3 条（最近/最成功/最戏剧性）
2. **推荐 Gene 网格**——3×3 卡片，按 category 分组，每张卡片有标题+描述+Install 按钮
3. **漏斗/进度条**——"你的 Agent 还没有参与进化 → 安装第一个 Gene → 跑一次 → 看结果"

最终选择：**组合方案**——上半部分是最近故事（有数据时）或引导文案（无数据时），下半部分是推荐 Gene 网格。

## 四、核心交互

### Q17. 用户点击一个 Gene 节点后想做什么？

**结论：** 看详情（策略步骤、成功率、谁在用）→ 决定是否安装。这与 Library tab 的功能重叠。Map 中的 Gene 点击应该直接跳转到 Library 的 Gene 详情。

### Q18. 用户点击一条边后想做什么？

**质疑：** 谁会点击一条线？现实中几乎没有用户知道边是可点击的。
**结论：** 边不需要独立的交互。hover 显示 tooltip 就够了。

### Q19. 搜索应该搜什么？

**结论：** 搜索在 Map 中没有意义。搜索应该在 Library tab 中做。

### Q20. hover 和 click 应该有什么区别？

**结论：**

- Hover：高亮关联路径 + tooltip（轻量，不改变页面状态）
- Click：跳转/打开详情面板（重量，改变页面状态）

### Q21. 缩放到底有没有必要？

**质疑：** 当前 4 级缩放是最大的过度设计。45 个 gene 不需要缩放——一屏可以放下。即使有 200 个 gene，力导向图+拖拽就够了。
**结论：** 不缩放。如果图放不下，用拖拽平移。如果节点太多（>100），用虚拟化（只渲染视口内的）。

**修改 MAP PRD：** 删除 4 级缩放。保留拖拽平移。

## 五、视觉与技术

### Q22. Canvas 是正确的技术选择吗？

**质疑：** Canvas 适合动画密集的场景。但如果我们改为卡片模式（冷启动），卡片用 HTML/CSS 更好。如果用图模式，Canvas 也可以，但力导向图用 SVG + d3-force 会更容易做交互（DOM 事件原生支持）。
**结论：** 自适应——卡片模式用 HTML，图模式用 Canvas。两套渲染器，一套数据源。

### Q23. 节点应该是什么形状和大小？

**结论：** 保留 Gene 形状编码（repair=六边形, optimize=圆, innovate=菱形），这是有意义的视觉区分。大小由 totalExecutions 驱动。

### Q24. 边应该是什么样子？

**结论：** 保留 confidence-to-color 编码（红→绿），线宽 by totalObs，虚线=探索前沿。这些都是好的设计。

### Q25. 背景应该是什么？

**结论：** 故事模式——普通页面背景。图模式——深色 + 点网格（现有设计）。

## 六、与其他 Tab 的关系

### Q26. Map 和 Library 有什么区别？

**结论：**

- **Map**：全局视角，展示关系和动态（"系统在学什么"）
- **Library**：个人视角，浏览和安装（"有什么可用的"）

Map 是看，Library 是做。

### Q27. Map 和 Feed 有什么区别？

**结论：**

- **Map**：空间布局，看关系和聚类
- **Feed**：时间线，看事件流

但在故事模式下，Map 和 Feed 会有重叠——都在展示最近发生了什么。区别在于 Map 的故事是精选的（最有代表性的 1-3 条），Feed 是完整的时间线。

### Q28. Map 和 My Evolution 有什么区别？

**结论：**

- **Map**：全局公开数据
- **My Evolution**：我的 Agent 的个人数据（需要登录）

## 七、长期演进

### Q29. 当有 500+ gene 和真实数据时，Map 应该变成什么样？

**结论：** 到那时 Graph 模式才有意义——可以看到 domain 聚类、热门路径、探索前沿。力导向图 + Louvain 社区检测 + 形状编码在数据充足时效果好。

### Q30. 超图（Hypergraph）改造后 Map 会变什么样？

**研究了什么：** SUPERGRAPH.md §6.1 提到因果追踪和维度保真。
**结论：** 超图数据可以让 Map 展示因果链——"这次成功是因为上次的失败学到了什么"。但这是 Phase 3+ 的事。当前优先级是让冷启动可用。

### Q31. 进化图谱最终会取代传统的 dashboard 吗？

**结论：** 不会。Dashboard（在 admin/analytics 中）服务运营者，Graph 服务开发者和访客。它们是互补的。

### Q32. Map 页面应不应该有"编辑"能力？

**结论：** 不应该。Map 是只读的展示层。编辑（创建 Gene、记录 outcome）在 SDK/CLI/My Evolution 中做。

## 八、综合结论

> **v1.0 回答后经过产品评审，以下结论已修正为 v2.0。**

### Q33. 基于以上分析，Map 页面的核心价值主张是什么？

**v1.0 结论（已否决）：** "让人一眼看到 Agent 进化正在发生。" → 太弱，一行统计数字就能做到。

**v2.0 结论：** "看到策略知识网络的形状和生长方向——哪里密集（成熟）、哪里稀疏（未探索）、哪里正在生长（活跃）。"

这是只有 Graph 能做的事。卡片做不了，列表做不了，数字做不了。**Graph 就是 Map 的身份。**

### Q34. 基于以上分析，Map 的冷启动策略应该是什么？

**v1.0 结论（已否决）：** "不画图画卡片" → 用 Library 的方案做 Map 的事，丧失独立存在的理由。

**v2.0 结论：** **始终画图。冷启动时在图上方叠加引导层。**

- Graph 始终存在（即使稀疏），是 Map 的身份
- 冷启动时叠加半透明引导卡片（精选 Gene + CTA）
- 引导卡片可以被关闭，关闭后看到完整的稀疏图
- 随着数据增长，图自然变丰富，引导层自然消失（capsules ≥ 10 不再显示）

### Q35. 基于以上分析，什么时候从"卡片模式"切换到"图模式"？

**v2.0 结论：** 不切换。没有"卡片模式"。始终是图模式。区别只在于：

- `capsules < 10`：图 + 引导叠加层
- `capsules ≥ 10`：图 + 故事横幅
- `capsules ≥ 100`：图充分展开，domain 聚类可见
